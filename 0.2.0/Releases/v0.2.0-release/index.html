Hello again, world! The Operator-Framework community is excited to announce the v0.2.0 release of operator-controller!

With this release of operator-controller, you can install an operator from a catalog of operators. This might sound familiar to exisiting users of OLM v0, however, there are some key differences that is evident in the user experience of installing an operator from a catalog of operators, even at this early stage of OLM v1.

Let us walk through an example of installing an operator from a catalog of operators, which will highlight the differences mentioned above.

First, let's get a Kind cluster fired up 

$ kind create cluster

Next, add a catalog of operators to the cluster to make the operators shipped in the catalog available for installation on the cluster. This is similar to how a repository needs to be added to the list of repositories available for a package manager to install from, when you want to install a binary available in the repository with the help of the package manager. We'll use the upstream community-operators catalog quay.io/operatorhubio/catalog: 

$ kubectl apply -f - <<EOF
apiVersion: catalogd.operatorframework.io/v1alpha1
kind: Catalog
metadata:
  name: operatorhubio
spec:
  source:
    type: image
    image:
      ref: quay.io/operatorhubio/catalog:latest
EOF

If we dig into the catalogd-system namespace, we can see our first difference here: 

$ kubectl get pods -n catalogd-system

operatorhubio                                  0/1     Completed   0          37s

Looks like there are no registry-pods running on cluster.

To install the argocd-operator, all we have to do is create an Operator CR: 

$ kubectl apply -f - <<EOF
apiVersion: operators.operatorframework.io/v1alpha1
kind: Operator
metadata:
  name: argocd-operator
spec:
  packageName: argocd-operator
EOF

$ kubectl get operators 
NAME              AGE
argocd-operator   53s

$  kubectl get operator argocd-operator -o yaml

To delete the operator, simply delete the Operator CR that was created earlier: 

$ kubectl delete operator argocd-operator 
operator.operators.operatorframework.io "argocd-operator" deleted
$ kubectl get namespaces | grep argocd
$ 
$ kubectl get crds | grep argo 
$

Notice again a key difference between OLM v0 and this v0.2.0 release of OLM when deleting an operator: besides the fact that deleting an installed operator is as easy as deleting the Operator CR, it also deletes every workload that was created during installation, including the CRDs

NOTE:

Installing specific versions of an operator

By default, the latest version of the operator is installed when a version is not specified in the Operator CR. We can confirm this (for now) by matching the image reference from the installedBundleResource field with the image reference for the latest version, which in argocd-operator's case, is v0.6.0:

```bash $ kubectl get bundlemetadata operatorhubio-argocd-operator.v0.6.0 -o yaml | grep image image: quay.io/operatorhubio/argocd-operator@sha256:1a9b3c8072f2d7f4d6528fa32905634d97b7b4c239ef9887e3fb821ff033fef6 - image: gcr.io/kubebuilder/kube-rbac-proxy:v0.8.0 - image: quay.io/argoprojlabs/argocd-operator@sha256:99aeec24cc406d06d18822347d9ac3ed053a702d8419191e4e681075fed7b9bb - image: quay.io/operatorhubio/argocd-operator@sha256:1a9b3c8072f2d7f4d6528fa32905634d97b7b4c239ef9887e3fb821ff033fef6

>
>Looks like the last image reference is in fact the one that is also visible in the `installedBundleResource` field. 
>
>We can also pin the installation of an operator to a specific version, if we wish to install something other than the latest:
